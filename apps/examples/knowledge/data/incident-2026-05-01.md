# Postmortem — Auth outage, 2026-05-01

**Severity:** SEV1
**Duration:** 41 minutes (09:12–09:53 UTC)
**Impact:** ~30% of API requests failed authentication; logins were unavailable.

Blameless postmortem, written per the [on-call runbook](runbooks/oncall.md).

## Summary

A deploy to the Auth service shipped a migration that added a `NOT NULL` column without a default.
The new pods crashed on boot reading the old rows, the rollout stalled half-deployed, and the
gateway's `/resolve` calls began timing out.

## Timeline (UTC)

| time | event |
|---|---|
| 09:08 | Auth deploy promoted to production |
| 09:12 | Error rate alert fires; primary on-call acknowledges |
| 09:21 | Identified the half-deployed rollout from the deploy dashboard |
| 09:34 | Rolled back per the [deploy runbook](runbooks/deploy.md) |
| 09:53 | Error rate back to baseline; incident closed |

## Root cause

The migration violated our **expand / contract** policy: the new column and the code reading it
shipped together, so the old image couldn't serve the new schema and the new image couldn't read the
old rows. Rollback was slower than it should have been because the migration wasn't backward-safe.

## What went well

- The alert fired fast and the rollout halted itself at 50% instead of taking down everything.

## Action items

- [ ] Add a CI check that rejects non-nullable columns without a default.
- [ ] Require migrations to be reviewed against the expand/contract checklist.
- [ ] Add a runbook drill for half-deployed rollouts to the next on-call onboarding.
